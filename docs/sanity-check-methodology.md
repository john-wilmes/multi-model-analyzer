# MMA Sanity Check Methodology

> **Note:** This methodology was written when tier-4 (Sonnet) and narration layers existed.
> Those features were removed in PR #88. Sections 2 and 6 are now historical reference only.

How we validate that MMA's analysis output matches reality. Each dimension has a
test corpus, a validation procedure, and accuracy metrics.

**Test Corpus**: Supabase ecosystem (10 repos: supabase monorepo, supabase-js,
ssr, auth-helpers, postgres-meta, storage, shared-types, embeddings-generator,
multiplayer.dev, pg-toolbelt). Indexed with `--enrich` flag for full tier-1
through tier-4 summarization + narrations. DB at
`data-supabase/mma-supabase.db`, mirrors at `data-supabase/mirrors/`.

---

## 1. Tier-3 Method Summaries (Haiku LLM)

**What**: Per-method summaries generated by Claude Haiku during `--enrich` indexing.

**How to test**:
1. Query DB: `SELECT value FROM kv WHERE key LIKE 'summary:tier3:%'` per repo
2. Sample 10+ methods across repos
3. Read actual source via `git show` from bare mirrors
4. Compare LLM summary against actual method behavior

**Metrics**: Accuracy rate (% of summaries that correctly describe the method).

**Last result**: ~60% accurate. Main failure mode: multi-line function signatures
truncated at tier-1 template level (`packages/summarization/src/templates.ts:34` reads only one line),
causing Haiku to receive incomplete context. Arrow function bodies leak into
return type descriptions (`packages/summarization/src/templates.ts:85`).

---

## 2. Tier-4 Service Summaries (Sonnet LLM)

**What**: Per-service architectural summaries generated by Claude Sonnet.

**How to test**:
1. Query DB: `SELECT value FROM kv WHERE key LIKE 'summary:tier4:%'` per repo
2. For each service summary, verify: (a) service name matches a real service,
   (b) described responsibilities match actual code, (c) listed dependencies
   are real
3. Check service catalog (`sarif:catalog:<repo>`) for over-segmentation

**Metrics**: Service identification accuracy, dependency accuracy.

**Last result**: 33 tier-4 summaries. 9 framework adapters misidentified as
separate services (over-segmentation). Dependencies field always `undefined`
(data collection bug). Real services described accurately when correctly scoped.

---

## 3. PageRank / Blast Radius

**What**: PageRank scores for import graph nodes; blast radius BFS for changed files.

**How to test**:
1. Overview API: `GET /api/blast-radius/:repo` returns ranked files
2. Verify top-ranked files are genuinely highly-imported (cross-reference with
   `grep -r "from.*<file>"` in mirror)
3. Detail API: `GET /api/blast-radius/:repo?file=<path>` returns affected files
4. Spot-check BFS results: are listed dependents actually importers?
5. Verify external packages (lodash, class-validator) are filtered from SARIF

**Metrics**: Top-5 precision, external package filtering correctness.

**Last result**: Not yet re-run against Supabase corpus. Previous run (Novu corpus) was mostly accurate: `@/` path aliases unresolved in one repo caused ~4x PageRank underestimation for aliased files. External package filtering works correctly. `directcss:` loader syntax created one phantom edge (now fixed via loader prefix stripping in `packages/structural/src/dependencies.ts`).

---

## 4. Cross-Repo Correlation

**What**: Edges connecting files across repos via npm package imports (e.g.,
supabase-js imports `@supabase/shared-types` from shared-types).

**How to test**:
1. Query: `SELECT count(*) FROM ... WHERE kind='imports' AND
   metadata.repo != target_repo`
2. Verify `packageRoots` map in `index-cmd.ts` resolves npm packages to repos
3. Sample cross-repo edges and verify in source: does repo A actually import
   the package published by repo B?
4. Count expected cross-repo edges from `package.json` dependencies

**Metrics**: Edge count vs expected, false positive rate.

**Last result**: **CRITICAL BUG** found (original corpus). `packageRoots` stored relative paths
(`packages/shared`) but `findRepoForPath` needed absolute paths
(`/path/to/mirror/packages/shared`). Result: 0 cross-repo edges resolved out of
~2,645 expected. Fix: `join(repoPath, dirname(pjFile.path))` at
`apps/cli/src/commands/indexing/phase-classify.ts:106` (moved from index-cmd.ts during refactor).

**Update (2026-03-20):** This bug has been fixed. Cross-repo edges now resolve correctly (349 cross-repo edges in Supabase corpus).

## 5. Search Quality (BM25/FTS5)

**What**: Full-text search over symbols, summaries, and file content.

**How to test**:
1. Run 6+ representative queries spanning different categories:
   - Structural: "what imports shared module"
   - Analytical: "most complex files"
   - Pattern: "repository pattern classes"
   - Synthesis: "summarize notification service"
   - Cross-repo: "cross-repo dependencies"
   - Temporal: "recent changes to auth"
2. Evaluate P@3 (precision at top 3 results)
3. Check query router classification accuracy

**Metrics**: P@3, routing accuracy per query type.

**Last result**: P@3 = 0.33 overall. Bimodal: perfect when router classifies
correctly, zero when misfire. Router misfires on 4/6 query types tested.
Structural and synthesis queries work; analytical, pattern, cross-repo, and
temporal fail.

---

## 6. Narration Accuracy

**What**: 13 repo-level LLM summaries (repo-arch, health, catalog per repo +
1 system narrative).

**How to test**:
1. Query: `SELECT value FROM kv WHERE key LIKE 'narration:%'`
2. For each narration, verify factual claims against DB and source:
   - File counts match `parsedFiles` KV
   - Service counts match `sarif:catalog:<repo>`
   - Pattern counts match `sarif:patterns:<repo>`
   - Dependency claims match import graph
3. Flag hallucinated specifics (invented numbers, non-existent patterns)

**Metrics**: Claim accuracy rate (verified/total claims per narration).

**Last result**: 8 accurate, 3 partially accurate, 4 inaccurate. All 4 catalog
narrations claim "no services" when DB has entries (prompt template bug). One
repo-arch narration fabricates pattern count (42 claimed, 52 actual — LLM
hallucination).

---

## 7. Fault Tree / Error Handling

**What**: SARIF findings for `fault/missing-error-boundary` and
`fault/unhandled-error-path`.

**How to test**:
1. Query: `SELECT value FROM kv WHERE key LIKE 'sarif:faultTree:%'`
2. Sample 5+ findings across repos
3. Read actual source for each flagged function
4. Classify: true positive (genuine missing error handling), false positive
   (intentional swallow, framework boundary, `console.error` not recognized)

**Metrics**: True positive rate from sample.

**Last result**: 209 findings (189 missing-error-boundary, 20 unhandled-error-path).
Sample of 5: 1 true positive, 4 false positives. Estimated ~20% true positive rate.
Root causes: `console.error` not recognized as logging, `.catch()` promise handlers
misidentified, intentional fallback patterns flagged, decorator/test code not
excluded.

---

## 8. Service-Call Edges (Queue + HTTP)

**What**: Edges representing inter-service communication via message queues
(BullMQ) and HTTP clients (fetch).

**How to test**:
1. Query graph for edges with `kind='queue'` or `kind='http'`
2. For each edge, read the source file and verify:
   - Queue: is it actually a BullMQ `.add()` / `.createWorker()` call?
   - HTTP: is it actually a `fetch()` call?
3. Check target resolution: does the target name match the actual queue/endpoint?
4. Flag in-process queues (p-queue) misidentified as message brokers

**Metrics**: True positive rate, target precision.

**Last result**: 34 edges total (18 queue, 16 HTTP). 33/34 are real code
patterns (97% precision). 1 false positive: `buffer.flushQueue.add()` in
`clickhouse-batch.service.ts` is a `p-queue` (in-process), not BullMQ.
13 edges have imprecise targets (generic `worker`, `_queue`, `external-api`).
1 test file included. 2 duplicate edges from same method.

---

## 9. Pattern Detection

**What**: Design pattern identification (Repository, Factory, Strategy, etc.)
via naming conventions and structural signatures.

**How to test**:
1. Query: `SELECT value FROM kv WHERE key LIKE 'sarif:patterns:%'`
2. For each detected pattern instance, verify in source:
   - Does the class/interface actually implement the pattern?
   - Is the naming convention match backed by structural evidence?
3. Check for false negatives: grep for known patterns not detected
   (e.g., `@Module`, `@Controller`, `@Injectable` in NestJS)

**Metrics**: Precision (true positives / detected), recall estimate.

**Last result**: Not yet re-run against Supabase corpus. Previous run (Novu corpus): 74 detections across 7 pattern types, 73/74 correct (98.6% precision). 1 marginal case. Major recall gap: NestJS decorator patterns (`@Module`, `@Controller`, `@Injectable`) entirely undetected — 400+ missed instances in that corpus.

---

## 10. Feature Flag Detection

**What**: Feature flags detected via enum patterns, `process.env` checks, and
LaunchDarkly `getFlag` calls.

**How to test**:
1. Query: `SELECT value FROM kv WHERE key LIKE 'flags:%'`
2. For each flag, verify in source:
   - Does the flag name match a real enum member or env var?
   - Is it used in a runtime conditional (if/ternary)?
3. Check for cross-repo double-counting (same flag defined in multiple repos)
4. Check for false positives (UI component enums, non-flag constants)

**Metrics**: Precision, cross-repo dedup accuracy.

**Last result**: Not yet re-run against Supabase corpus. Previous run (Novu corpus): 75 flags detected, ~98.7% precision (74/75). Main issues: UI component enum members misclassified as feature flags, and cross-repo double-counting for flags shared across repos via a common enum.

---

## 11. Circular Dependency Detection

**What**: Import cycles detected via Tarjan's SCC algorithm on the import graph.

**How to test**:
1. Query: `SELECT value FROM kv WHERE key LIKE 'circularDeps:%'`
2. For each cycle, verify in source:
   - Do the listed files actually import each other?
   - Is it a barrel-file-mediated cycle (index.ts re-exports creating false
     cycle) or a genuine mutual dependency?
3. Classify: real mutual dependency vs. barrel artifact

**Metrics**: True positive rate, barrel-mediated fraction.

**Last result**: Not yet re-run against Supabase corpus. Previous run (Novu corpus): 127 cycles detected, 105/127 (83%) real circular dependencies, 22/127 (17%) barrel-mediated artifacts. The `suppressBarrelCycles` option (added in PR #76) addresses the barrel artifact class. Recommend enabling it to reduce false positives.

---

## 12. Dead Export Detection

**What**: Exported symbols with no importers within the analyzed codebase.

**How to test**:
1. Query: `SELECT value FROM kv WHERE key LIKE 'sarif:deadExports:%'`
2. Sample detected dead exports
3. Verify in source: is the export genuinely unused, or consumed via:
   - Path aliases (`@/`, `~`)
   - Dynamic imports (`import()`)
   - External consumers outside the analyzed repos
   - Re-exports through barrel files

**Metrics**: True positive rate (genuinely dead vs. alias/dynamic false positives).

**Last result**: Agent timed out scanning `node_modules` in bare mirrors. The
grep-based validation approach is impractical for dead exports because:
1. Bare repos include `node_modules/` with massive source maps
2. Path aliases (`@/`) and dynamic imports make static grep unreliable
3. External consumers outside the analyzed repos are invisible

Recommended test approach: instead of grep, use the import graph edges already in
the DB. A dead export is one where `symbol.exported = true` AND no import edge
targets it. Compare against manual spot-checks of 10 exports.

## Running the Full Sanity Check

```bash
# 1. Index with enrichment
rm -f data-supabase/mma-supabase.db
ANTHROPIC_API_KEY=sk-ant-... node apps/cli/dist/index.js index \
  -c data-supabase/mma-supabase.config.json --enrich -v

# 2. Start dashboard for interactive verification
node apps/cli/dist/index.js dashboard --db data-supabase/mma-supabase.db

# 3. Run automated checks (examples)
node -e "
const Database = require('better-sqlite3');
const db = new Database('data-supabase/mma-supabase.db');

// Count by category
for (const prefix of ['summary:tier3:', 'summary:tier4:', 'sarif:patterns:',
  'sarif:faultTree:', 'flags:', 'circularDeps:', 'narration:']) {
  const count = db.prepare(
    \"SELECT count(*) as n FROM kv WHERE key LIKE ?\"
  ).get(prefix + '%');
  console.log(prefix + '*:', count.n);
}
"
```
